# Sync — Синхронизация и публикация

Скилл управляет ветками в двухуровневой модели:

```
dev   ← рабочая ветка, все мерджат сюда
main  ← прод, обновляется ТОЛЬКО через CMS (кнопка «Опубликовать») или Valery вручную
  ├── valery-sayfullin
  ├── stacy
  └── 533va
```

## Определение пользователя

Получи имя из git:
```bash
git config user.name
```

Нормализуй в имя ветки: lowercase, пробелы → дефисы, убрать спецсимволы.
Пример: `Valery Sayfullin` → `valery-sayfullin`

## Режим 1: Начало работы

Вызывай когда пользователь начинает работу или в начале сессии.

### Шаги

1. Определи пользователя (git user.name)
2. Получи свежее состояние:
   ```bash
   git fetch origin
   ```
3. Проверь есть ли ветка пользователя (`git branch --list <имя>`)
4. Если ветки нет — создай от свежего dev:
   ```bash
   git checkout dev && git pull origin dev && git checkout -b <имя>
   ```
5. Если ветка есть — переключись и обнови поверх dev (rebase, не merge!):
   ```bash
   git checkout <имя>
   git pull origin <имя> --rebase 2>/dev/null || true
   git rebase origin/dev
   ```
6. Если при rebase возникли конфликты — запусти визард конфликтов (см. ниже), затем `git rebase --continue`

### Почему rebase, а не merge

`git pull origin dev --no-edit` создаёт мусорные merge-коммиты в ветке пользователя («Merge branch dev into stacy»). Rebase ставит коммиты пользователя поверх свежего dev — история остаётся чистой и линейной.

### Вывод

```
🔄 ═══════════════════════════════════════
   SYNC · НАЧАЛО РАБОТЫ
═══════════════════════════════════════
👤 Пользователь ................ Valery Sayfullin
🌿 Ветка ....................... valery-sayfullin
📥 Dev ......................... ✅ Синхронизирован
═══════════════════════════════════════
   Готово! Работай в своей ветке.
   Когда закончишь — скажи «опубликуй»
═══════════════════════════════════════
```

## Режим 2: Публикация в dev

Вызывай когда пользователь говорит: «опубликуй», «мерж», «в dev», «готово, публикуй», «sync publish».

### Шаги

1. Проверь что все изменения закоммичены. Если нет — предложи закоммитить
2. Запомни имя ветки пользователя
3. Получи свежий dev:
   ```bash
   git fetch origin dev
   git checkout dev
   git pull origin dev
   ```
4. Мержи ветку пользователя:
   ```bash
   git merge <ветка> --no-edit
   ```
5. Если конфликтов нет:
   ```bash
   git push origin dev
   ```
6. Если конфликты — запусти визард конфликтов, потом push
7. После успешного мержа — вернись на ветку пользователя и обнови:
   ```bash
   git checkout <ветка>
   git rebase origin/dev
   ```

### Вывод при успехе

```
🚀 ═══════════════════════════════════════
   SYNC · ПУБЛИКАЦИЯ В DEV
═══════════════════════════════════════
👤 Пользователь ................ Valery Sayfullin
🌿 Ветка ....................... valery-sayfullin → dev
📦 Коммиты ..................... 3 коммита влиты
🔄 Push ........................ ✅ Отправлено в dev
═══════════════════════════════════════
   Опубликовано в dev!
   Проверь на preview.dvizh.cc
═══════════════════════════════════════
```

## Режим 3: Публикация в main — ОГРАНИЧЕНО

Если пользователь говорит «в main», «мерж в main», «опубликуй в прод»:

### Проверка доступа

Получи имя пользователя из `git config user.name`.

Если НЕ содержит «valery» (регистронезависимо):
```
⛔ ═══════════════════════════════════════
   Публикация в main доступна только Valery.
   Используй «опубликуй» для мержа в dev.
═══════════════════════════════════════
```
Прекрати выполнение.

Если Valery — предупреди:
```
⚠️ Публикация в main (прод)
   Рекомендуется собирать через кнопку «Опубликовать» в CMS.
   Она автоматически соберёт из dev, провалидирует и задеплоит.

   Мержить вручную? (да/нет)
```

Если подтвердил:
```bash
git checkout main && git pull origin main
git merge dev --no-edit
git push origin main
git checkout dev
```

## Визард конфликтов

Когда merge или rebase даёт конфликты — НЕ решай молча. Проведи пользователя через каждый файл.

### Для каждого конфликтного файла:

1. Покажи понятное описание:
```
⚠️ Конфликт в файле: src/components/ui/Form.astro

📄 Версия из dev (коллега, 25 фев 14:30):
   Изменён стиль кнопки — добавлен hover-эффект

📄 Твоя версия (26 фев 10:15):
   Добавлено новое поле «телефон»

Что делаем?
  1) Оставить версию из dev (коллеги)
  2) Оставить твою версию
  3) Объединить оба изменения (я попробую автоматически)
```

2. Дождись ответа пользователя
3. Примени выбор
4. Переходи к следующему файлу

### Определение дат и авторов

Для показа информации о конфликтующих изменениях:
```bash
git log origin/dev --oneline -1 -- <файл>
git log <ветка> --oneline -1 -- <файл>
```

### После разрешения всех конфликтов

Для merge:
```bash
git add . && git commit --no-edit
```

Для rebase:
```bash
git add . && git rebase --continue
```

Продолжи с push как в обычном режиме.

## Важно

- НИКОГДА не делай force push в dev или main
- Force push в ветку пользователя допустим ТОЛЬКО после rebase (и это нормально)
- НИКОГДА не удаляй чужие ветки
- При сомнениях — спрашивай пользователя
- Ветка пользователя НЕ удаляется после мержа — она постоянная
- main — только через CMS или Valery вручную
