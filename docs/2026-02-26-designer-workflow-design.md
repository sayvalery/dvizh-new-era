# Воркфлоу дизайнера: качество кода и компонентов

Дата: 2026-02-26

## Контекст

Сайт dvizh.io строится на чистом стеке: Astro + Tailwind + Alpine.js. Никаких тяжёлых фреймворков. Дизайнеры будут добавлять компоненты через Claude Code, используя экспорт из Figma/Pencil.dev. Нужно сохранить чистоту проекта при масштабировании команды.

Текущее состояние: 13 компонентов в трёх папках (layout/, ui/, blocks/), рантайм JS — только Alpine.js (~15KB), стили — Tailwind inline + global.css.

## Проблемы, которые решаем

1. **Новые зависимости** — кто-то может добавить React, Framer Motion, UI-библиотеку
2. **Дублирование компонентов** — создаётся новый Button когда уже есть похожий
3. **Некосистентность стилей** — разные отступы, цвета, паттерны
4. **Регрессии** — сломали существующее при добавлении нового

## Решение: два инструмента

### 1. CLAUDE.md — конституция проекта

Файл CLAUDE.md в корне проекта. Claude автоматически читает его в начале каждой сессии.

**Содержание:**

**Стек и ограничения:**
- Разрешённые рантайм-зависимости: Alpine.js, Tailwind CSS. Точка. Всё остальное — спрашивать у пользователя.
- Компоненты — только Astro (.astro), никакого React, Vue, Svelte
- CSS: Tailwind-классы inline для компонентов. Общие стили (анимации, утилиты, переходы) — в `global.css` через `@layer`. Один файл для глобального, не размазывать по проекту.

**Каталог компонентов:**
Живой список всех компонентов с описанием. Обновляется при каждом добавлении. Перед созданием нового — обязательно проверить каталог и реальные файлы в `apps/web/src/components/`.

**Конвенции:**
- Именование: PascalCase, .astro расширение
- Структура: `layout/` (обёртки), `ui/` (переиспользуемые), `blocks/` (рендерят CMS-данные)
- Props через TypeScript interface
- Интерактивность: Alpine.js (x-data, x-show, @click). Inline vanilla JS для простых случаев.

**Процесс интеграции (обязательный при любых изменениях компонентов/страниц):**

Шаг 1: Анализ входящего — что это (страница, секция, компонент), какие элементы внутри.

Шаг 2: Сопоставление с существующими — проверить каталог в CLAUDE.md и файлы в `components/`. Показать пользователю что переиспользуется, что создаётся новое.

Шаг 3: План интеграции — что переиспользуется, что создаётся (и почему), какие файлы затронуты. Подтверждение перед работой.

Шаг 4: Реализация — код по конвенциям. Обновить каталог компонентов если создан новый.

Шаг 5: Автоматический `/frontend-check` — Claude запускает скилл проверки после завершения работы.

### 2. Скилл `/frontend-check`

Вызывается автоматически после работы с фронтенд-компонентами и страницами.

**Проверки:**
- Зависимости: нет новых в package.json
- Каталог: новые компоненты внесены в CLAUDE.md
- Билд: Astro собирается без ошибок
- Список затронутых страниц для визуальной проверки

## Флоу дизайнера

```
1. Дизайнер создаёт макет в Figma / Pencil.dev
2. Экспортирует код или описывает словами в Claude
3. Claude анализирует (шаг 1-2 процесса интеграции)
4. Claude показывает план: "Переиспользую PersonCard, создаю новый HeroSection"
5. Дизайнер подтверждает
6. Claude реализует (шаг 4)
7. Claude автоматически запускает /frontend-check (шаг 5)
8. Дизайнер проверяет визуально по списку URL
```

## Что НЕ входит

- Storybook / pixel-perfect проверки — overengineering для текущего масштаба
- Pre-commit hooks — лишняя инфраструктура
- Визуальная регрессия (скриншоты) — можно добавить позже если понадобится
