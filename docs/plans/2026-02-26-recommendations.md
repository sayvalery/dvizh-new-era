# Рекомендации по улучшению проекта

## Для стабильности

### 1. Оптимизация изображений (Astro Image)
Сейчас все `<img>` теги используют прямые URL без оптимизации. Astro имеет встроенный компонент `<Image>` который автоматически:
- Конвертирует в WebP/AVIF
- Генерирует srcset для responsive
- Добавляет width/height для CLS
- Lazy loading

**Результат:** Lighthouse Performance +15-25 баллов, экономия трафика 40-60%.

### 2. Fallback для CMS-контента при сборке
Если CMS недоступна в момент сборки — билд падает. Нужно:
- Кэшировать последний успешный ответ API в `.cache/`
- При ошибке CMS использовать кэш
- Предупреждать в логе о stale data

### 3. TypeScript типы для CMS
Все CMS-данные типизированы как `any`. Payload CMS v3 генерирует типы автоматически:
```bash
npx payload generate:types
```
Это даст автокомплит и проверку полей на этапе разработки.

### 4. Мониторинг 404 ошибок
Добавить простой скрипт проверки внутренних ссылок при сборке — `astro-link-check` или аналог. Это предотвратит битые ссылки.

## Для упрощения работы

### 5. Preview-режим для маркетологов
Сейчас чтобы увидеть изменения контента, нужен `pnpm build`. Добавить:
- Webhook из Payload CMS на автоматический re-deploy
- Preview-среду на отдельном домене (staging.dvizh.io)

### 6. Компонент `<Picture>` для контент-менеджеров
Обёртка которая принимает CMS-медиа и автоматически:
- Добавляет alt из CMS
- Обрабатывает отсутствие картинки (серый блок)
- Оптимизирует размеры

### 7. Visual Editor через Payload Live Preview
Payload CMS v3 поддерживает Live Preview — маркетолог видит страницу в реальном времени при редактировании. Нужно настроить `livePreview` в конфиге коллекций.

### 8. Шаблоны страниц в CMS
Вместо хардкода продуктовых страниц в `.astro` — создать коллекцию `pages` в Payload с блок-редактором. Маркетолог сможет сам собирать страницы из компонентов (Hero, Features, Stats, CTA) без разработчика.

## Для дизайнера

### 9. Design Tokens в Figma → Tailwind
Настроить автоматическую синхронизацию дизайн-токенов:
- Figma Variables → JSON → tailwind.config.ts
- Один источник правды для цветов, шрифтов, отступов

### 10. Storybook-подобный UI Kit
Текущий `/ui-kit` работает хорошо. Можно улучшить:
- Добавить варианты props через URL-параметры
- Добавить секцию "Spacing/Layout"
- Документировать Guidelines (когда какой компонент)

## Для CEO: аргументы за свой стек

### Преимущества перед Tilda/Taptop:

| Критерий | Tilda/Taptop | Свой стек (Astro + Payload) |
|---|---|---|
| **Скорость загрузки** | 2-5 сек (JS-фреймворк, трекеры) | < 1 сек (статика, ~15KB JS) |
| **SEO** | Ограничен шаблонами | Полный контроль: JSON-LD, sitemap, meta |
| **Стоимость хостинга** | ~$50-200/мес за тарифы | ~$15/мес (VPS + CDN) |
| **Вендор-лок** | Полный. Уходить дорого | Нулевой. Open-source стек |
| **Кастомизация** | Ограничена виджетами | Безграничная |
| **Производительность на масштабе** | Деградирует (500+ страниц) | Не деградирует (статика) |
| **Безопасность** | Зависит от платформы | Статика = минимальная поверхность атаки |
| **Интеграции** | Через iframe/JS | Нативные (API, webhooks) |

### Ключевой аргумент:
> "511 страниц генерируются за 10 секунд. Весь JS на клиенте — 15KB (Alpine.js).
> Lighthouse Performance 95+. Стоимость хостинга — $15/мес. Ни одна конструкторная платформа не может предложить такое."

### ROI:
- Время разработки страницы: ~2 часа (компоненты переиспользуются)
- Изменение контента: 15 минут в CMS + автодеплой
- Lighthouse: 95+ vs 60-70 на Tilda
- Нет абонентской платы за количество страниц
