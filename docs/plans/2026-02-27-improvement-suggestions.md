# План улучшений dvizh.io -- 27 февраля 2026

Комплексный план по улучшению проекта dvizh.io (Astro 5 SSG + Payload CMS v3).
Каждый пункт содержит описание, пользу и оценку трудозатрат.

---

## 1. Стабильность и DevOps

### 1.1 Кэширование CMS-ответов при сборке
**Проблема:** Если CMS недоступна в момент `pnpm build` -- билд падает, сайт не обновляется. Таймаут всего 2 секунды (`payload.ts`), при сетевых задержках это может привести к ложным ошибкам.

**Решение:**
- При успешном запросе к CMS сохранять ответ в `.cache/cms/` (JSON-файлы по endpoint + params hash)
- При ошибке CMS -- использовать кэш и выводить предупреждение в лог
- Добавить env-переменную `CACHE_CMS=true|false` для управления

**Польза:** Билд никогда не падает из-за временной недоступности CMS. Маркетологи могут деплоить когда CMS на обслуживании.
**Трудозатраты:** Средние (4-6 часов). Изменения в `payload.ts` -- обёртка вокруг `fetchFromCMS`.

---

### 1.2 Расширенный post-deploy smoke test
**Проблема:** Текущий `build-smoke.sh` проверяет только наличие файлов и плейсхолдеры. Не проверяет реальную работу форм и CMS API.

**Решение:** Расширить `build-site.sh`:
- После билда проверить: главная (200 OK), контакты, блог
- Проверить что `<form>` присутствует в HTML ключевых страниц
- Тестовый POST на `/api/form-submissions` с пометкой `test: true`
- При провале -- не переключать nginx на новую версию (blue-green deploy)
- Результат в Telegram через Bot API

**Польза:** Автоматическое откатывание при сломанном билде. Мгновенное уведомление о проблемах.
**Трудозатраты:** Средние (3-5 часов). Bash-скрипт + Telegram Bot Token.

---

### 1.3 Периодический health check (cron)
**Проблема:** Если CMS упадёт между деплоями -- формы перестанут работать, но никто не узнает пока клиент не пожалуется.

**Решение:**
- Cron-задача каждые 5 минут: `GET /api/blog-posts?limit=1` + `POST /api/form-submissions` (тестовый)
- При ошибке -- Telegram уведомление
- Простой bash-скрипт ~30 строк, без дополнительных зависимостей

**Польза:** Узнаём о поломках в течение 5 минут, а не от клиентов.
**Трудозатраты:** Низкие (1-2 часа). Один bash-скрипт + строка в crontab.

---

### 1.4 Blue-green deploy для статики
**Проблема:** Во время билда (~10 секунд) nginx может раздавать частично обновлённые файлы.

**Решение:**
- Билдить в `dist-new/`, а не в `dist/`
- После успешного smoke test -- атомарно переключить симлинк: `ln -sfn dist-new dist`
- Сохранять предыдущую версию в `dist-prev/` для быстрого отката

**Польза:** Нулевой даунтайм при деплое. Возможность моментального отката.
**Трудозатраты:** Низкие (2-3 часа). Изменения в `build-site.sh` и `nginx.conf`.

---

### 1.5 Логирование и мониторинг
**Проблема:** Логи deploy-webhook.js идут только в stdout контейнера. Нет централизованного журнала деплоев.

**Решение:**
- Добавить файл `logs/deploy.log` с ротацией (logrotate или по размеру)
- Записывать: дата, триггер, длительность билда, результат smoke test, количество страниц
- Простая страница `/admin/deploys` в CMS для просмотра истории

**Польза:** Можно понять когда и что ломалось. Удобно для отладки.
**Трудозатраты:** Низкие (2-3 часа).

---

### 1.6 Бэкап базы данных
**Проблема:** PostgreSQL хранит весь контент CMS. При потере данных -- потеря всех статей, глоссария, лидов.

**Решение:**
- Cron-задача: ежедневный `pg_dump` в `backups/` с ротацией (хранить 7 последних)
- Еженедельная копия на внешнее хранилище (S3/Yandex Object Storage)
- Скрипт восстановления из бэкапа

**Польза:** Защита от потери данных. Возможность восстановления за минуты.
**Трудозатраты:** Низкие (2-3 часа). Bash + cron.

## 2. Производительность

### 2.1 Оптимизация изображений
**Проблема:** Все `<img>` используют прямые URL из CMS без оптимизации. Нет WebP/AVIF, нет srcset, нет lazy loading для below-the-fold контента.

**Решение:**
- Создать компонент `CmsImage.astro` -- обёртка для изображений из CMS
- Использовать Payload CMS Image Sizes (уже есть sharp в зависимостях) для генерации нескольких размеров на стороне CMS
- Настроить `imageSizes` в коллекции Media для генерации: thumbnail (400px), medium (800px), large (1200px), og (1200x630)
- В компоненте: `<picture>` с `<source>` для разных размеров, `loading="lazy"` для below-the-fold

**Польза:** Lighthouse Performance +15-25 баллов. Экономия трафика 40-60%. Улучшение Core Web Vitals (LCP).
**Трудозатраты:** Средние (4-6 часов). Новый компонент + изменения в Media.ts.

---

### 2.2 Оптимизация шрифтов
**Проблема:** Шрифты подключены как TTF (~500KB+ суммарно). Нет preload. Нет subset для кириллицы.

**Решение:**
- Конвертировать TTF в WOFF2 (экономия 30-50% на размере)
- Добавить `<link rel="preload">` для критичных шрифтов (Inter Regular, Styrene A Medium)
- Создать subset только для кириллицы + латиницы (убрать неиспользуемые глифы)
- Рассмотреть `font-display: optional` для Styrene A (для LCP)

**Польза:** Уменьшение загрузки шрифтов на 50-70%. Улучшение FCP и LCP.
**Трудозатраты:** Низкие (2-3 часа). Конвертация + изменения в `global.css` и `BaseLayout.astro`.

---

### 2.3 Critical CSS и выше-линии-сгиба
**Проблема:** Весь CSS загружается одним файлом. Для первого экрана достаточно малой части.

**Решение:**
- Astro уже хорошо делает tree-shaking CSS, но можно помочь:
- Добавить `<link rel="preload" as="style">` для основного CSS
- Убедиться что Astro генерирует scoped стили для компонентов (уже работает)
- Рассмотреть `astro-critical-css` для инлайнинга критического CSS

**Польза:** Faster First Contentful Paint на медленных соединениях.
**Трудозатраты:** Низкие (1-2 часа). Минимальные изменения.

---

### 2.4 Кэширование статики через Cloudflare
**Проблема:** Текущий nginx кэширует статику, но только на уровне сервера. CDN не используется.

**Решение:**
- Cloudflare Tunnel уже настроен (cloudflared в docker-compose.prod.yml)
- Настроить Page Rules: `*.css`, `*.js`, `*.woff2` -- кэш 1 год
- Включить Brotli-сжатие на уровне Cloudflare
- Настроить Cache-Control headers в nginx с `immutable` для хэшированных файлов (уже частично есть)

**Польза:** Глобальное ускорение загрузки. Разгрузка сервера. Бесплатно на Cloudflare Free.
**Трудозатраты:** Низкие (1-2 часа). Настройки в Cloudflare Dashboard.

---

### 2.5 Предзагрузка критических ресурсов
**Проблема:** В `<head>` нет hints для браузера о приоритетных ресурсах.

**Решение:** Добавить в `BaseLayout.astro`:
```html
<link rel="preload" href="/fonts/inter-variable.woff2" as="font" type="font/woff2" crossorigin>
<link rel="preload" href="/fonts/styrene-a-medium.woff2" as="font" type="font/woff2" crossorigin>
<link rel="dns-prefetch" href="https://cdn.dvizh.io">
```

**Польза:** Ускорение отрисовки шрифтов на 200-400ms.
**Трудозатраты:** Низкие (30 минут). 3-4 строки в BaseLayout.astro.

## 3. Контент-менеджмент

### 3.1 Коллекция "Страницы" в CMS (Page Builder)
**Проблема:** Продуктовые страницы (ipoteka, vitrina и т.д.) захардкожены в `.astro` файлах. Маркетолог не может менять текст без разработчика.

**Решение:**
- Создать коллекцию `Pages` в Payload CMS с блоковым редактором
- Блоки: Hero, Features (FeaturesGrid), Stats (StatsSection), CTA, Testimonials, FAQ, Timeline
- Каждый блок -- отдельный Payload Block с типизированными полями
- Маршрут `apps/web/src/pages/[...slug].astro` для рендеринга CMS-страниц
- Продуктовые страницы постепенно переводить из хардкода в CMS

**Польза:** Маркетологи самостоятельно создают и редактируют страницы. Меньше задач для разработчиков.
**Трудозатраты:** Высокие (2-3 дня). Новая коллекция + 6-8 блоков + маршрут.

---

### 3.2 Preview-среда для маркетологов
**Проблема:** Чтобы увидеть результат изменений, нужен полный билд. Маркетолог не может быстро проверить как выглядит статья.

**Решение:**
- preview.dvizh.cc уже настроен (nginx proxy на dev-сервер :4321)
- Довести до ума: Astro dev-сервер берёт данные напрямую из CMS в реальном времени
- Кнопка "Предпросмотр" в CMS (уже настроен `livePreview` для коллекций blog-posts, persons, companies, glossaries)
- Добавить preview для остальных коллекций: videos, research, cases

**Польза:** Маркетолог видит результат в реальном времени. Уменьшение количества "пустых" деплоев.
**Трудозатраты:** Низкие (2-3 часа). Расширить livePreview в payload.config.ts.

---

### 3.3 Автоматический slug из заголовка
**Проблема:** Маркетолог вводит slug вручную. Может допустить ошибку, забыть транслитерацию.

**Решение:**
- Payload CMS хук `beforeChange`: автоматическая генерация slug из `title` при создании
- Транслитерация кириллицы (обязательно для SEO-friendly URL)
- Slug остаётся редактируемым, но заполняется автоматически

**Польза:** Меньше ошибок. Ускорение работы маркетолога.
**Трудозатраты:** Низкие (1-2 часа). Хук + функция транслитерации.

---

### 3.4 Уведомление в Telegram о новых лидах
**Проблема:** Сейчас лиды из форм уходят только в webhook (Albato/n8n). Если webhook не настроен -- лиды лежат молча в базе.

**Решение:**
- В `FormSubmissions.ts` afterChange хук: помимо webhook, отправлять в Telegram
- Формат: имя, компания, email, телефон, страница, пресет формы
- Env-переменная `TELEGRAM_BOT_TOKEN` и `TELEGRAM_CHAT_ID`

**Польза:** Моментальное оповещение о новых заявках. Не зависит от настройки Albato.
**Трудозатраты:** Низкие (1-2 часа). ~20 строк кода в хуке.

---

### 3.5 Шаблоны контента в CMS
**Проблема:** При создании новой статьи маркетолог начинает с пустой страницы.

**Решение:**
- Создать 3-4 шаблона статей в CMS: "Продуктовый обзор", "Кейс", "Новость", "Экспертное мнение"
- Каждый шаблон -- предзаполненная структура блоков (H1, Intro, Image, Body, CTA)
- Payload CMS поддерживает дублирование документов -- использовать как основу

**Польза:** Ускорение создания контента. Единообразие структуры статей.
**Трудозатраты:** Низкие (2-3 часа). Создание шаблонных документов в CMS.

## 4. SEO

### 4.1 Структурированные данные (JSON-LD) для всех типов страниц
**Проблема:** JSON-LD реализован только для blog-posts (`BlogPosting`). Продуктовые страницы, главная, компании -- без структурированных данных.

**Решение:**
- **Главная:** `Organization` + `WebSite` с `potentialAction: SearchAction`
- **Продуктовые страницы:** `Product` или `SoftwareApplication`
- **О нас:** `Organization` с contactPoint, address, foundingDate
- **Контакты:** `ContactPage` + `Organization`
- **Глоссарий:** `DefinedTerm` или `FAQPage`
- **Видео:** `VideoObject`
- **Персоны:** `Person` с `worksFor`

**Польза:** Расширенные сниппеты в Google. Улучшение CTR из поиска на 15-30%.
**Трудозатраты:** Средние (4-6 часов). JSON-LD объекты для каждого типа страницы.

---

### 4.2 Расширенные мета-теги
**Проблема:** Не все страницы имеют уникальный description. Нет `article:published_time` для блога. Нет `og:image` для продуктовых страниц.

**Решение:**
- Проверить и заполнить `<meta name="description">` для всех статических страниц
- Для блога добавить: `article:published_time`, `article:modified_time`, `article:author`, `article:section`
- Создать OG-изображения для каждой продуктовой страницы (1200x630)
- Добавить `<meta name="author">` для статей

**Польза:** Лучшее отображение в соцсетях. Корректная индексация дат статей.
**Трудозатраты:** Средние (3-4 часа).

---

### 4.3 Улучшение sitemap
**Проблема:** Автогенерируемый sitemap от `@astrojs/sitemap` не содержит `<lastmod>`, `<changefreq>`, `<priority>`.

**Решение:**
- Настроить кастомные опции в `astro.config.ts` для sitemap:
  ```ts
  sitemap({
    changefreq: 'weekly',
    priority: 0.7,
    lastmod: new Date(),
    filter: (page) => !page.includes('/ui-kit'),
  })
  ```
- Исключить `/ui-kit`, `/thanks`, `/404` из sitemap
- Добавить `<lastmod>` из `updatedAt` CMS для динамических страниц

**Польза:** Поисковики точнее индексируют актуальный контент. Исключение служебных страниц.
**Трудозатраты:** Низкие (1-2 часа). Конфигурация + фильтрация.

---

### 4.4 Хлебные крошки (Breadcrumbs) с микроразметкой
**Проблема:** Хлебные крошки есть только в блоге, без JSON-LD разметки.

**Решение:**
- Создать компонент `Breadcrumbs.astro` с JSON-LD `BreadcrumbList`
- Использовать на всех внутренних страницах (продукты, решения, блог, глоссарий)
- Автоматическое построение пути: Главная > Продукты > Ипотека

**Польза:** Расширенные сниппеты в Google с навигацией. Улучшение навигации пользователей.
**Трудозатраты:** Низкие (2-3 часа). Один компонент + интеграция.

---

### 4.5 Каноникализация и редиректы
**Проблема:** Нет обработки trailing slash. Нет редиректов со старого домена Webflow.

**Решение:**
- Убедиться что `trailingSlash` в `astro.config.ts` установлен явно
- Настроить 301 редиректы в nginx для старых URL Webflow (если есть база URL)
- Добавить `<link rel="canonical">` с trailing slash (уже есть в BaseLayout)

**Польза:** Избежание дублирования страниц в индексе. Сохранение SEO-веса со старого сайта.
**Трудозатраты:** Средние (2-4 часа). Зависит от количества старых URL.

## 5. Developer Experience

### 5.1 TypeScript типы для CMS-данных
**Проблема:** Все CMS-данные типизированы как `any` (`docs: any[]`). Нет автокомплита, легко ошибиться в имени поля.

**Решение:**
- Payload CMS v3 автоматически генерирует TypeScript типы: `npx payload generate:types`
- Файл `payload-types.ts` уже настроен в `payload.config.ts`
- Импортировать типы в `payload.ts` и компоненты: `import type { BlogPost, Person } from '@cms/payload-types'`
- Заменить `any` на конкретные типы

**Польза:** Автокомплит в IDE. Ошибки ловятся при компиляции, а не на проде.
**Трудозатраты:** Средние (3-4 часа). Генерация типов + замена `any` в ~15 файлах.

## 6. Бизнес-кейс для CEO: почему Astro + Payload лучше Tilda/Taptop

### 6.1 Скорость и производительность

**Tilda/Taptop:**
- Загрузка страницы: 2-5 секунд (тяжёлый JS-фреймворк, трекеры платформы, render-blocking ресурсы)
- JavaScript на клиенте: 300-800KB (jQuery, анимации, виджеты платформы)
- Lighthouse Performance: 40-70 баллов
- При 500+ страницах платформа начинает тормозить (редактор, публикация, загрузка)

**Наш стек (Astro + Payload):**
- Загрузка страницы: < 1 секунда (статический HTML, никаких JS-фреймворков)
- JavaScript на клиенте: ~15KB (только Alpine.js для форм и меню)
- Lighthouse Performance: 90-100 баллов
- 511 страниц генерируются за ~10 секунд. Производительность не деградирует с ростом

**Вывод:** Наш сайт загружается в 3-5 раз быстрее. Это напрямую влияет на конверсию -- Google доказал, что каждая дополнительная секунда загрузки снижает конверсию на 7%.

### 6.2 SEO-превосходство статических сайтов

**Tilda/Taptop:**
- SEO ограничен шаблонами платформы (нельзя добавить произвольный JSON-LD)
- Дублирование контента (технические URL платформы)
- Медленная загрузка = штраф от Google (Core Web Vitals)
- Ограниченный контроль над meta-тегами, canonical, hreflang
- Нет программного управления sitemap (все страницы в один файл)

**Наш стек:**

- Чистые URL без технического мусора платформы
- Статика = максимальные баллы Core Web Vitals = приоритет в ранжировании
- Программное управление sitemap с lastmod, priority, changefreq
- Возможность A/B тестов на уровне мета-тегов

**Вывод:** Статический сайт получает приоритет в поиске Google. По данным Google, сайты с Lighthouse > 90 получают заметное преимущество в ранжировании перед конкурентами на конструкторах.

## 7. Дизайн-система

### 7.1 Design Tokens: Figma -> Tailwind
**Проблема:** Цвета и шрифты определены отдельно в Figma и в `tailwind.config.ts`. При изменении -- нужно обновлять в двух местах.

**Решение:**
- Экспортировать Design Tokens из Figma как JSON (Figma Variables -> Tokens Studio)
- Скрипт `scripts/sync-tokens.js` который конвертирует JSON в `tailwind.config.ts`
- Один источник правды: дизайнер меняет в Figma, запускает скрипт, токены обновляются

**Польза:** Автоматическая синхронизация дизайна. Нет рассинхронизации Figma и кода.
**Трудозатраты:** Средние (4-6 часов). Скрипт + настройка Tokens Studio.

---

### 7.2 Компонентная карта для дизайнера
**Проблема:** Дизайнер не знает какие компоненты уже реализованы и что они умеют.

**Решение:**
- Расширить UI Kit страницу с чётким маппингом: "Figma Frame -> Astro Component"
- Для каждого компонента: скриншот из Figma + живой пример на сайте
- Документировать допустимые вариации (props), чтобы дизайнер проектировал в рамках системы
- Текущие компоненты:
  - `NavBar` (sticky, dropdown, mobile drawer)
  - `Form` (4 пресета x 3 лейаута = 12 вариантов)
  - `CTA` (3 варианта: banner, inline, floating)
  - `PersonCard` (2 варианта: inline, full)
  - `BentoGrid`, `FeaturesGrid`, `StatsSection`, `HeroSection`, `Timeline`
  - `SuccessStoriesCarousel`, `SuccessStory`

**Польза:** Дизайнер проектирует быстрее. Меньше "кастомных" компонентов. Консистентность.
**Трудозатраты:** Средние (3-4 часа). Документация + скриншоты.

---

### 7.3 Spacing и Typography Scale
**Проблема:** Нет чёткой системы отступов. Каждая страница может использовать разные `py-16`, `py-24`, `py-32`.

**Решение:**
- Определить фиксированные "секционные" отступы: `section-sm` (py-12), `section-md` (py-16), `section-lg` (py-24)
- Определить типографическую шкалу: какие размеры шрифтов для заголовков, подзаголовков, текста
- Задокументировать в UI Kit и как Tailwind-утилиты в `tailwind.config.ts`:
  ```ts
  spacing: {
    'section-sm': '3rem',
    'section-md': '4rem',
    'section-lg': '6rem',
  }
  ```

**Польза:** Визуальный ритм сайта. Дизайнер знает что использовать.
**Трудозатраты:** Низкие (2-3 часа). Конфиг + документация.

---

### 7.4 Процесс добавления нового компонента
**Проблема:** Процесс описан в CLAUDE.md, но нет чеклиста для дизайнера.

**Решение:** Формализовать чеклист:

1. Дизайнер создаёт макет в Figma
2. Проверяет: можно ли реализовать существующим компонентом? (UI Kit)
3. Если нет -- описывает Claude: "Нужен компонент X, вот Figma-ссылка"
4. Claude анализирует, предлагает план (переиспользование vs новый)
5. Дизайнер подтверждает
6. Claude реализует + добавляет в UI Kit
7. `/frontend-check` -- автоматическая проверка
8. Дизайнер проверяет визуально по URL

**Польза:** Чёткий workflow. Меньше хаоса при масштабировании команды.
**Трудозатраты:** Низкие (1 час). Документация.
