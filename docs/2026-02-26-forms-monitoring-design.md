# Формы и мониторинг: дизайн системы надёжности

Дата: 2026-02-26

## Контекст

Сайт dvizh.io — статический (Astro SSG), формы отправляются клиентским fetch через nginx на Payload CMS (`POST /api/form-submissions`). Маркетологи будут самостоятельно добавлять контент и триггерить ребилд сайта. Нужно чтобы система работала автономно, без постоянного контроля со стороны отдела дизайна.

Боль из прошлого: на Webflow формы ломались "тихо" — не было никакого способа узнать что что-то отвалилось, пока клиент не пожалуется.

## Принципы

- **Никакого тяжёлого фронта** — сайт статический, без localStorage, без retry-очередей
- **"Сделал и забыл"** — настроить один раз, дальше работает без вмешательства
- **Быстро узнавать о поломках** — алерты в Telegram
- **Автономность маркетинга** — публикуют контент, видят результат, не зовут разработчиков

## Архитектура: три уровня защиты

### Уровень 1: Пайплайн публикации с уведомлениями

Маркетолог нажимает кнопку "Опубликовать на сайт" в CMS (не afterChange, а отдельная кнопка) → ребилд сайта → smoke test → результат в Telegram + в интерфейс CMS.

Кнопка реализуется как кастомный React-компонент в Payload admin + Next.js API route (`/api/deploy`) внутри CMS. Отдельный webhook-listener не нужен — CMS сама является оркестратором.

Результат в интерфейсе CMS:
```
✅ Билд завершён
✅ Smoke test: главная страница (200 OK)
✅ Smoke test: формы найдены на /contacts
✅ Smoke test: POST /api/form-submissions (200 OK)
✅ Telegram: уведомление отправлено
→ Сайт обновлён
```

Или при ошибке:
```
✅ Билд завершён
❌ Smoke test: форма не найдена на /contacts
→ Сайт НЕ обновлён, работает старая версия
→ Уведомление отправлено в Telegram
```

### Уровень 2: Post-deploy smoke test

Автоматически после каждого ребилда:
- Ключевые страницы отдаются (200 OK)
- Формы присутствуют в HTML
- POST на `/api/form-submissions` проходит
- Если не прошло → nginx остаётся на старой версии, алерт в Telegram

### Уровень 3: Периодический health check

Cron на том же сервере, каждые 5 минут:
- CMS жива? (GET `/api/globals/navigation`)
- Формы принимаются? (POST test на `/api/form-submissions`)
- Нет → Telegram

## Инфраструктура

Всё на одном сервере, никаких дополнительных серверов:

```
Сервер
├── Docker Compose
│   ├── postgres
│   ├── cms (Payload + кнопка деплоя + API route /api/deploy)
│   ├── nginx (статика + проксирование)
│   └── cloudflared (опционально)
│
└── cron (встроен в ОС)
    └── Каждые 5 мин → health-check.sh → если плохо → Telegram
```

Отдельный webhook-listener не нужен — CMS (Next.js) сама обрабатывает запрос на деплой через свой API route.

## Оценка сложности

| Компонент | Сложность | Влияние на сайт |
|-----------|-----------|-----------------|
| Post-deploy smoke test | Низкая (расширить build-site.sh, ~50-100 строк bash) | Ноль, +5-10 сек к деплою |
| Кнопка деплоя в CMS | Средняя (React-компонент + Next.js API route + Telegram бот) | Ноль, всё в фоне |
| Периодический health check | Низкая (cron + bash скрипт ~20 строк) | Ничтожное (1 GET + 1 POST каждые 5 мин) |

## Stage vs Production

Два окружения:
- **Stage (dev-сервер)** — CMS + динамический Astro dev (или свежий билд). Маркетолог проверяет контент тут.
- **Production (основной домен)** — только статика через nginx. Обновляется только по кнопке "Опубликовать на сайт".

Stage закрыт от индексации через nginx Basic Auth (пароль). Поисковики не видят, случайные посетители не зайдут.

## Флоу маркетолога

```
1. Пишет статью в CMS (/admin)
2. Нажимает стандартный "Publish" в Payload → статья видна на stage
3. Заходит на stage, проверяет как выглядит
4. Всё ок → нажимает "Опубликовать на сайт" (кастомная кнопка)
5. CMS запускает: билд → smoke test → результат в интерфейс + Telegram
6. Через ~минуту статья на проде
```

## Кастомизация Payload CMS

Payload v3 — code-first CMS на Next.js. Кастомные компоненты — штатный механизм:
- React-компоненты в `admin.components` конфига
- Next.js API routes для серверной логики
- Не конфликтует с обновлениями Payload (кастомный код в `apps/cms/src/`, Payload в `node_modules/`)
