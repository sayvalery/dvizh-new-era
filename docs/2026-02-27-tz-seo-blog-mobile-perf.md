## Задача 2: Блог и контент-хаб — миграция legacy-контента

### Текущее состояние

#### Структура данных в CMS

141 пост в коллекции `blog-posts`. Два пути рендеринга контента:

1. **`content` (blocks)** — новый формат (Lexical rich text, блоки image/cta/form/video/quote). **Используется 1 постом** (тестовый `buy-2026`)
2. **`bodyHtml` (textarea)** — сырой HTML из Webflow. **Используется 139 постами** (98.6%)
3. 1 пост пустой (podcast-1)

#### Проблемы с bodyHtml

Webflow HTML содержит:
- Пустые атрибуты `id=""` на каждом элементе
- Кастомные Webflow-классы: `b-article-card`, `data-rt-embed-type`
- Встроенные Webflow-компоненты без CSS → рендерятся как бесстильные блоки
- Ссылки на `dvizh.webflow.io` и `www.dvizh.io` — перезаписываются в `[slug].astro`, но только для `src`/`href`
- Нет стилей для этих Webflow-блоков в текущем `global.css`

#### Рендеринг

Страница `blog/[slug].astro` (строка 127-129):
```
post.content?.length > 0
  ? <BlockRenderer blocks={post.content} />
  : bodyHtml && <div class="prose prose-lg max-w-none" set:html={bodyHtml} />
```

BodyHtml рендерится через Tailwind `prose` — базовая типографика работает, но Webflow-специфичные элементы выглядят криво.

#### Неполный Lexical-сериализатор

`RichTextBlock.astro` — кастомный сериализатор Lexical. **Не поддерживает:**
- Блоки кода, inline code
- Strikethrough
- Таблицы
- Горизонтальные линии
- Upload/image ноды
- Indent

#### Legacy-поля

- `author` (text) — **мёртвое поле**, 0 постов его используют. Описание в админке: "use primaryAuthor". Фронтенд имеет фоллбэк на него.

#### Качество данных

- Пробелы в начале названий: " Правила девелопера" (категория), " Алексей Балыбердин" (автор)
- Все 8 категорий имеют `order: 0` — сортировка не работает
- Тег "Mega menu" (slug: mega-menu) — системный тег, не контентный

#### Дублирование кода

`blog/index.astro` и `blog/page/[page].astro` содержат почти идентичный код (фильтры категорий, сетка постов, лейаут).

#### Размеры изображений

CMS генерирует 3 размера: `thumbnail` (400×300), `card` (768×480), `cover` (1200×630). **Фронтенд использует только базовый `url`** — ни `sizes.card.url`, ни `sizes.thumbnail.url` не применяются.

### Два варианта решения

#### Вариант А: Конвертировать bodyHtml → blocks (полная миграция)

**Плюсы:** единый формат, полный контроль, чистый рендеринг
**Минусы:** очень трудоёмко (139 постов), нужен парсер HTML→Lexical, риск потери контента

**Оценка:** 139 постов × анализ HTML + конвертация + проверка. Нужен скрипт-мигратор. Потенциально 2-3 дня работы.

#### Вариант Б: Улучшить рендеринг bodyHtml (прагматичный путь) ← рекомендация

**Плюсы:** быстро, без риска, обратно совместимо
**Минусы:** два пути рендеринга остаются навсегда

**Шаги:**
1. Добавить CSS-стили в `global.css` для Webflow-специфичных классов и элементов
2. Написать функцию `sanitizeBodyHtml()` — убрать пустые `id=""`, Webflow-классы, конвертировать кастомные блоки (цитаты, карточки) в семантический HTML
3. Новые посты писать через blocks (Lexical)
4. Старые посты остаются в bodyHtml, но с улучшенным рендерингом

**Оценка:** 1 день работы.

## Задача 4: Аудит и оптимизация производительности

### Текущие метрики

| Метрика | Значение | Оценка |
|---------|----------|--------|
| CSS бандл | 57 KB / 9.6 KB gzip | Отлично |
| JS бандл (Alpine) | 45 KB / 16.7 KB gzip | Нормально |
| JS бандл (MeshGradient) | 21 KB / 6.7 KB gzip — только главная | Нормально |
| Шрифты (5 файлов TTF) | **2.5 MB** (3 preload = 1.4 MB) | Плохо |
| Сторонние скрипты | 0 | Отлично |
| Dist (511 страниц) | 19 MB | Нормально |

### Найденные проблемы

#### Высокий приоритет

**1. Шрифты в формате TTF вместо WOFF2**

5 шрифтов по 277-883 KB каждый, суммарно 2.5 MB. WOFF2 даёт сжатие 20-30%, экономия ~500-750 KB.

| Файл | TTF | WOFF2 (оценка) |
|------|-----|----------------|
| inter-variable.ttf | 854 KB | ~600 KB |
| inter-italic-variable.ttf | 883 KB | ~620 KB |
| styrene-a-medium.ttf | 276 KB | ~190 KB |
| styrene-a-bold.ttf | 277 KB | ~195 KB |
| styrene-a-black-italic.ttf | 284 KB | ~200 KB |

**Действие:** конвертировать все шрифты в WOFF2, обновить `@font-face` в `global.css` и `preload` в `BaseLayout.astro`.

**2. MeshGradient — WebGL шейдер на главной**

Полноэкранный WebGL canvas с `requestAnimationFrame` — постоянная нагрузка на GPU. Библиотека `@paper-design/shaders` (21 KB).

**Что проверить:**
- Автоматически ли шейдер ставится на паузу, когда hero уходит из viewport (IntersectionObserver)?
- Можно ли снизить `speed` или использовать `reduced-motion` media query?
- CSS-фоллбэк градиент уже есть — для слабых устройств можно показывать только его

**Действие:** добавить IntersectionObserver — останавливать шейдер когда hero не видно. Добавить `prefers-reduced-motion` → показывать только CSS-градиент.

**3. LogoStream — бесконечная анимация blur**

7 элементов одновременно анимируют `filter: blur()` от 0 до 10px + `transform` + `opacity`. Каждый с `will-change: transform, opacity, filter`. Blur — одна из самых дорогих CSS-операций.

**Что проверить:**
- Можно ли заменить blur на opacity? Визуальный эффект "глубины" можно имитировать через scale + opacity без blur
- Можно ли уменьшить количество одновременных элементов?
- `will-change` на 7 элементах резервирует 7 GPU-слоёв

**Действие:** попробовать убрать `filter: blur()` из анимации, оставить только `transform` + `opacity`. Если визуал устраивает — огромный выигрыш в производительности.

#### Средний приоритет

**4. NavBar backdrop-blur на скролле**

`backdrop-blur-2xl` (40px blur) на фиксированном навбаре — при каждом кадре скролла браузер блюрит всё за навбаром.

**Действие:** заменить `backdrop-blur-2xl` на `backdrop-blur-md` (12px) или `backdrop-blur-lg` (16px). Визуально почти идентично, но в 2-3 раза дешевле для GPU.

**5. Три font preload = 1.4 MB**

Preload: `inter-variable.ttf` (854 KB) + `styrene-a-bold.ttf` (277 KB) + `styrene-a-medium.ttf` (276 KB). Все три качаются параллельно с первым запросом.

**Действие:** оставить preload только для `inter-variable` (основной шрифт текста). Heading-шрифты загружать через обычный `@font-face` — они не критичны для first paint.

**6. Нет srcset / responsive images**

CMS генерирует 3 размера (thumbnail 400×300, card 768×480, cover 1200×630), но фронтенд всегда грузит полный URL.

**Действие:** в `BlogCard.astro` и других компонентах использовать `srcset` с CMS-размерами.

**7. Hero-изображение без loading/fetchpriority**

`HeroSection.astro` строка 104 — above-the-fold изображение без `loading="eager"` и `fetchpriority="high"`.

**Действие:** добавить оба атрибута.

#### Низкий приоритет

**8. inter-italic-variable.ttf (883 KB)** — объявлен но редко используется. Не preloaded (хорошо), но скачивается при первом появлении italic текста в CMS-контенте.

**9. Homepage HTML = 73 KB** — мега-меню HTML дублируется на каждой странице. Не критично для статического сайта.

### Чеклист оптимизации

- [x]Конвертировать все .ttf → .woff2
- [x]Обновить `@font-face` и preload на .woff2
- [x]Убрать preload для heading-шрифтов (оставить только Inter)
- [x]MeshGradient: добавить IntersectionObserver для паузы вне viewport
- [x]MeshGradient: `prefers-reduced-motion` → только CSS-градиент
- [x]LogoStream: убрать `filter: blur()`, оставить `transform` + `opacity`
- [x]LogoStream: убрать `will-change: filter` (оставить `transform, opacity`)
- [x]NavBar: `backdrop-blur-2xl` → `backdrop-blur-lg`
- [x]HeroSection: `loading="eager"` + `fetchpriority="high"` на hero-картинку
- [x]BlogCard: добавить `srcset` с CMS image sizes
- [ ] Запустить Lighthouse аудит до и после оптимизации *(требует браузер)*
- [ ] Проверить на реальном мобильном устройстве *(ручная проверка)*
