## Вариант 1: Astro + Strapi (или аналог) на своем хостинге

**Суть:** статический сайт на Astro, контент в headless CMS (Strapi, Directus, или даже Tina), хостинг на российском VPS.

**Плюсы:**
- Полный контроль над инфраструктурой. Заблокировали — перенесли за час.
- Маркетологи получают нормальный редактор в CMS, публикуют сами.
- Статика = скорость, SEO, простота деплоя.
- AI-инструменты реально ускоряют сборку компонентов на Astro, это хорошо ложится на задачу.
- Стоимость хостинга — пара тысяч рублей в месяц вместо $400.
- Формы — просто компоненты с конфигом эндпоинта, меняются в одном месте.

**Минусы:**
- Первоначальная настройка инфраструктуры потребует DevOps.
- Вам двоим придется освоить базовый workflow: git, деплой, структуру Astro-проекта. Не rocket science, но порог входа есть.
- Любые нестандартные хотелки от маркетинга (новый тип блока, необычная страница) — это код, а не drag-and-drop.

## Вариант 2: Российский конструктор (Taptop / Тильда) + отдельный хостинг через проксирование

**Суть:** остаетесь в конструкторе, но ставите перед ним свой nginx/CDN на российском сервере, который проксирует трафик.

**Плюсы:**
- Минимум изменений в рабочем процессе.
- Быстро поднять.

**Минусы:**
- Вы описали патовую ситуацию с VPN — проксирование ее не решает полностью, потому что ресурсы конструктора (скрипты, шрифты, изображения) все равно грузятся с их серверов.
- Зависимость от чужой платформы остается.
- По сути это костыль, а не решение.

## Вариант 3: WordPress на своем хостинге

**Суть:** классика. Полный контроль, огромная экосистема.

**Плюсы:**
- Маркетологи точно умеют с ним работать.
- Полный контроль хостинга.
- Тысячи готовых решений для форм, SEO, всего.

**Минусы:**
- Требует поддержки: обновления, безопасность, плагины ломаются.
- Для кастомного дизайна нужен разработчик или возня с конструкторами тем.
- Медленнее, тяжелее, больше точек отказа, чем статика.
- AI-инструменты хуже помогают с WordPress-темами, чем с чистым фронтендом.

2. **Решение — уйти на self-hosted стек**: Astro + Payload CMS на российском VPS решает корневую проблему контроля инфраструктуры. Для корпоративного сайта с ~100 статьями и глоссарием Astro с его островной архитектурой — правильный выбор. Payload CMS вместо Strapi дает лучшую интеграцию с TypeScript и более гибкую кастомизацию.

Но нужно учесть нюансы: Payload v3 построен на Next.js, поэтому требует отдельного Node.js сервера, хотя это не критично. Payload использует Lexical для rich text и предоставляет REST API и GraphQL, которые Astro может потреблять. Монорепа структура поможет управлять зависимостями между фронтом и CMS.

Развертывание будет раздельным — Payload как серверное приложение, Astro генерирует статику, Docker Compose упростит оркестрацию. Главное, что я не упомянул — Payload v3 сильно завязан на Next.js, и это может быть ограничением в некоторых сценариях.

Payload требует больше ресурсов, чем Strapi, потому что запускает Next.js для админки — нужен VPS с минимум 2GB RAM, лучше 4GB. Сборка статики для 200-300 страниц не проблема, Astro справится за пару минут, но если масштабироваться до тысяч страниц, это может замедлиться. Но начальная настройка Payload с collections, blocks и access control — это TypeScript код, что сложнее чем просто HTML с переменными, хотя AI хорошо генерирует такие конфиги. Docker на VPS — стандартный подход, но требует DevOps для настройки, обновлений и мониторинга, что не бесплатно по времени. При миграции нужно сохранить URL-структуру для SEO.

**Что я недосказал или приукрасил:**

**1. Payload v3 работает поверх Next.js.** Админка Payload — это Next.js приложение. Это значит VPS нужен не слабый: минимум 2-4GB RAM. Не критично по деньгам, но нужно учитывать при выборе сервера.

**2. Начальная настройка — не "пару дней DevOps".** Docker + Postgres + Nginx + CI/CD пайплайн + бэкапы + SSL + CORS + rate limiting + DNS. Реалистично — неделя DevOps-работы, может полторы.

**3. SEO при миграции.** Мы вообще не обсудили. Если у вас есть органический трафик, при переезде нужно: сохранить URL-структуру один в один, настроить 301-редиректы для изменившихся URL, проверить sitemap, переподключить Search Console. Иначе потеряете позиции.

**4. Миграция контента.** 100 статей и 100 глоссарий не перенесутся сами. Нужен скрипт: Webflow API отдает контент, скрипт заливает в Payload. Автоматизируемо, но это отдельная задача на день-два.

**5. VPN-проблема наоборот.** Мы хостим на российском VPS. Люди с зарубежным VPN будут видеть его? Как правило, да — российские серверы из-за рубежа доступны. Проблема с Taptop скорее всего была в их специфичной CDN/DNS настройке. Но это стоит протестировать до полного переключения.

**6. DNS — отдельная тема.** Нужен российский DNS-провайдер (Selectel, Yandex Cloud DNS), иначе рискуем получить ту же проблему с блокировкой DNS, от которой убегаем.

**7. Я слишком легко сказал "AI напишет компонент за час".** Для простых блоков — да. Для чего-то с нетривиальной логикой (калькулятор с хитрой формулой, мультишаговая форма с условиями) может потребоваться фронт. Ты сам сказал — в крайнем случае фронты помогут. Это нормально, просто не стоит продавать CEO идею, что фронты вообще никогда не понадобятся.
