# Предлагаемые улучшения FSD

### **1. Введение слоя `domains`**

#### **Описание**

Вводится новый верхнеуровневый слой `domains`, который располагается между `app` и `pages`. Каждый слайс в этом слое представляет собой **ограниченный контекст (Bounded Context)** из DDD и объединяет связанные части бизнес-логики (например, `user`, `partner-solutions`, `mortgage`). Внутренняя структура каждого домена (`domains/user`) повторяет классическую структуру FSD и может включать в себя собственные `pages`, `widgets`, `features`, `entities` и `shared`, относящиеся только к этому домену.

#### **Влияние на архитектуру**

* **Четкая декомпозиция:** Приложение логически разделяется на независимые бизнес-вертикали. Это упрощает навигацию по кодовой базе и снижает когнитивную нагрузку на разработчиков.
* **Инкапсуляция бизнес-логики:** Каждый домен инкапсулирует свою логику и данные. Взаимодействие между доменами происходит исключительно через их публичные API (`index.ts`), что обеспечивает низкую связанность.
* **Подготовка к микрофронтендам:** Такая структура является естественным шагом к микрофронтендной архитектуре. При необходимости любой домен может быть относительно легко выделен в самостоятельное приложение.

#### **Обоснование**

Введение слоя `domains` — это стратегическое решение для управления сложностью в крупных монолитных приложениях. Оно позволяет масштабировать кодовую базу, избегая хаоса, и способствует параллельной работе нескольких команд над разными бизнес-задачами, не создавая конфликтов.

### **2. Паттерн `Repository` для взаимодействия с API**

#### **Описание**

Все **CRUD-операции (Create, Read, Update, Delete)** с сущностями предметной области инкапсулируются с помощью паттерна `Repository`. Файлы репозиториев размещаются в сегменте `api` соответствующей сущности (`entities/my-entity/api/repository.ts`). API-методы, не относящиеся к CRUD (например, `send-sms-code`), реализуются как отдельные утилиты в том же сегменте (`entities/my-entity/api/send-sms-code.ts`).

#### **Влияние на архитектуру**

* **Абстракция от источника данных:** Репозиторий становится единой точкой доступа к данным сущности, скрывая детали реализации (Apollo GQL, REST API, localStorage). Компоненты и сервисы не знают, откуда приходят данные.
* **Разделение ответственности:** Логика получения и преобразования данных четко отделена от бизнес-логики и UI, что соответствует принципу **Single Responsibility Principle (SRP)**.

#### **Обоснование**

* **Тестируемость:** Код, использующий репозитории, становится легко тестируемым. В тестах реальный репозиторий подменяется mock-объектом, что избавляет от необходимости делать реальные сетевые запросы.
* **Гибкость и поддерживаемость:** При смене библиотеки для работы с API (например, переход с Apollo на `fetch`) или изменении формата данных, правки потребуются только внутри репозитория, а не во всем приложении.
* **Чистота кода:** Компоненты и Pinia-сторы освобождаются от логики API-запросов, становясь более декларативными и сфокусированными на своих прямых задачах.

-----

### **3. Строгий контроль потока данных**

#### **Описание**

Взаимодействие с API и последующее изменение состояния приложения происходит по строго регламентированному сценарию:

1.  **Изменение состояния:** Вызов метода API, который должен изменить глобальное состояние, происходит **только внутри `actions` в Pinia-сторах**.
2.  **Получение данных без изменения состояния:** Прямое обращение к репозиториям или API-утилитам из компонентов возможно только для операций, которые не меняют состояние в Pinia (например, получение данных для формы, которая не сохраняется в сторе).

#### **Влияние на архитектуру**

* **Предсказуемость:** Поток данных становится строго однонаправленным: `Компонент -> Action (Pinia) -> Repository (API) -> Mutation (Pinia) -> State -> Компонент`. Это упрощает отладку и понимание работы приложения.
* **Централизация логики:** Вся логика, связанная с управлением состоянием, концентрируется в сторах Pinia, которые становятся единственным источником правды.

#### **Обоснование**

Этот подход предотвращает хаотичное разбрасывание API-запросов по компонентам, что является частой причиной дублирования кода и усложняет поддержку. Усиление принципа **однонаправленного потока данных (Unidirectional Data Flow)** делает архитектуру более надежной и отказоустойчивой.

-----

### **4. Разделение сущностей на два типа**

#### **Описание**

Все сущности (`entities`) негласно разделяются на два типа в зависимости от того, где находится "источник правды" для их данных.

1.  **Backend-driven сущности:** Данные, для которых **бэкенд является единственным источником правды** (например, `User`, `Product`). Они всегда используют репозитории для CRUD-операций. Их состояние в Pinia-сторе помечается как `readonly`, чтобы предотвратить случайные изменения данных на клиенте в обход API.
2.  **Frontend-only сущности:** Данные, для которых **фронтенд является источником правды** (например, состояние UI-модального окна, данные формы). Они не имеют сегмента `/api` и репозитория. Их состояние полностью управляется в Pinia-сторе.

#### **Влияние на архитектуру**

* **Два типа Pinia-сторов:** Появляются два четко определенных типа сторов с разными правилами работы с состоянием (одни — с `readonly state`, другие — с полностью управляемым).
* **Явные правила:** Для каждого типа сущности существует свой набор правил, что упрощает принятие архитектурных решений и делает код более предсказуемым.

#### **Обоснование**

* **Отражение реальности:** Такое разделение точно отражает роль фронтенда как "тонкого клиента", где большая часть бизнес-логики и правил валидации находится на бэкенде.
* **Иммутабельность:** Использование `readonly` защищает критически важные данные от непреднамеренных мутаций, что особенно важно при работе с GQL, который по умолчанию возвращает иммутабельные объекты.

### **5. Разрешение кросс-импортов для `entities`**

#### **Описание**

В виде исключения из правил FSD разрешается кросс-импорт сущностей.
Для этого рекомендуется использовать специальный алиас `@x`, чтобы сделать такие импорты явными.

Для этого в структуре директорий сущности, для которой требуется импорт другой сущности, требуется создать директорию с именем `@x`.

Пример:
Сущность `Client` включается в себя сущность `ClientProfile`.

Тогда структура директорий будет выглядеть следующим образом - `/src/domains/user/entities/client/@x/user/client-profile.ts`.
Где в `client-profile.ts` мы импортируем и потом экспортируем части сущности `ClientProfile` от которых зависит `Client`.
Обычно это модели, тип и трансформеры.

#### **Влияние на архитектуру**

* **Снижение дублирования кода:** Позволяет избежать дублирования TypeScript-типов и интерфейсов, создавая единый источник правды для моделей данных.
* **Точная типизация агрегатов:** Дает возможность корректно описывать агрегированные сущности, которые часто возвращает GraphQL (например, `User` с вложенным `Profile`).

#### **Обоснование**

Классические правила FSD могут быть излишне строгими при работе с GraphQL. Разрешение импорта типов — это прагматичное послабление, которое не нарушает ключевых принципов изоляции, но значительно повышает качество и надежность кодовой базы за счет строгой и переиспользуемой типизации.

# Ключевые особенности FSD о которых всегда надо помнить

### **1. Публичный API слайсов (`index.ts`)**

#### **Описание**

Публичный API — это файл `index.ts`, расположенный в корне каждого **слайса** (например, `features/auth/by-password/index.ts`). Он служит единственной "точкой входа" для импорта любой части этого слайса извне. Все, что не экспортируется из этого файла, считается приватной частью реализации слайса.

#### **Влияние на архитектуру**

* **Контроль зависимостей:** Создается четкая граница между "внутренностями" слайса и остальным приложением. Это предотвращает хаотичные импорты и нарушение правила "слои могут обращаться только к слоям, лежащим ниже".
* **Инкапсуляция и гибкость:** Внутренняя структура слайса полностью инкапсулируется. Если потребуется реорганизовать файлы внутри слайса, изменения затронут только его внутренние импорты и один файл — `index.ts`. Код, использующий этот слайс, останется неизменным.

#### **Обоснование**

Публичный API — это фундаментальный инструмент для создания управляемой архитектуры. Он работает как **контракт**, который слайс предоставляет внешнему миру. Это делает рефакторинг безопасным и предсказуемым, а также позволяет большой команде работать над проектом, не ломая код друг друга.

### **2. Матрица зависимостей FSD 2.1**

Эта матрица показывает, какие слои могут импортировать (`From`) другие слои (`To`). Поток зависимостей всегда однонаправленный — от верхних слоев к нижним.

| Из ⟶ В | `app` | `pages` | `processes` | `widgets` | `features` | `entities` | `shared` |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: | :---: |
| **`app`** | ❌ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| **`pages`** | ❌ | ❌ | ✅ | ✅ | ✅ | ✅ | ✅ |
| **`processes`** | ❌ | ❌ | ❌ | ✅ | ✅ | ✅ | ✅ |
| **`widgets`** | ❌ | ❌ | ❌ | ❌ | ✅ | ✅ | ✅ |
| **`features`** | ❌ | ❌ | ❌ | ❌ | ❌ | ✅ | ✅ |
| **`entities`** | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | ✅ |
| **`shared`** | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |

* **✅** — Импорт **разрешён**.
* **❌** — Импорт **запрещён**.

### **Пояснения к матрице**

* **`app`**: Самый верхний слой, он собирает всё приложение, поэтому может импортировать все остальные слои.
* **`pages`**: Отвечает за страницы и их композицию, поэтому может использовать `processes`, `widgets` и `features`, а также `entities` и `shared`.
* **`widgets`**: Собирает сложные блоки UI из `features`, `entities` и `shared`.
* **`features`**: Инкапсулирует бизнес-логику, которая работает с `entities` и общими утилитами из `shared`.
* **`entities`**: Содержит модели данных и логику, которая может использовать только `shared` (например, общие типы).
* **`shared`**: Самый независимый слой, он ничего не импортирует из слоев, находящихся выше.

#### **Как воспринимать правило матрицу зависимостей FSD и конкретное ограничение, что shared нельзя импортировать в shared?**

В слое `shared` нет **слайсов**.

Слайсы — это разделение кода по бизнес-сущностям/бизнес-действиям. Слой `shared`, по определению, содержит код, который **не зависит от бизнес-логики** проекта, поэтому делить его на слайсы не имеет смысла.

Вместо слайсов слой `shared` делится напрямую на **сегменты**. Каждый сегмент — это, по сути, независимый модуль или "библиотека" с кодом определённого назначения.

Это правило касается разных сегментов слоя `shared`. К примеру у нас есть сегмент `/shared/ui`, допускается импорт из `/shared/ui` в `/shared/ui/`, но нельзя импортировать в другие сегменты.

#### Основные сегменты слоя `shared`

Вот стандартный набор сегментов для `shared` и их предназначение:

* `ui` 🧱
    * **Что это:** Переиспользуемые, "глупые" UI-компоненты, из которых строится весь интерфейс. Ваш UI-kit.
    * **Примеры:** `ButtonControl`, `InputContorol`, `CheckboxControl`, `Modal`, `Spinner`, `Logo`.

* `lib` 🧰
    * **Что это:** Вспомогательные функции, утилиты и хуки, не связанные с UI или API. Ваш "ящик с инструментами".
    * **Примеры:** функция `formatDate`, хелпер `classNames`, кастомный хук `useDebounce`, валидаторы.

* `api` 🔌
    * **Что это:** Конфигурация API-клиента (например, `axios instance`), общие функции для запросов и обработчики ошибок.
    * **Примеры:** `axiosInstance`, `baseQuery` для RTK Query, общие функции-адаптеры для данных.

* `model` 📦
    * **Что это:** Общие для всего приложения типы, сторы (Zustand, Redux) и хуки для работы с состоянием, не привязанные к конкретной бизнес-сущности.
    * **Примеры:** тип `type TColor = 'red' | 'blue'`, стор для управления темой (светлая/тёмная), селекторы для этого стора.

* `config` ⚙️
    * **Что это:** Глобальные константы, переменные окружения и конфигурации.
    * **Примеры:** `API_URL`, роуты (`ROUTES_PATH`), константы для i18n, feature-флаги.

Таким образом, `shared` — это просто набор независимых друг от друга папок (сегментов), каждая из которых решает свою задачу.

### **3. Что я создаю?**

При добавлении новой функциональности всегда задавайте себе следующие вопросы:

* **Переиспользуемую утилиту или "глупый" UI-компонент вне конкретной предметной области или без контекста?** → `shared`.
* **Переиспользуемую утилиту или "глупый" UI-компонент для конкретной предметной области или контекста?** → `domains/*/shared`.
* **Часть предметной области?** (`User`, `Partner Solutions`) → `domains`.
* **Являются ли эти данные частью общей бизнес-логики (как «Пользователь» или «Заявка»), которую для создания или изменения необходимо отправлять на сервер, чтобы он как единственный источник правды сохранил состояние?** → `entities` (Backend-driven сущность).
* **Являются ли эти данные временным состоянием интерфейса, которое существует только в браузере и может быть безболезненно утеряно при перезагрузке страницы, источником правды для которого ялвяется текущее состоние фронта?** → `entities` (Frontend-only сущность).
* **Действие, которое пользователь совершает для получения выгоды?** (`phone-verification`, `create-draft-application`) → `features`.
* **Сложный, составной блок интерфейса из нескольких фич и сущностей?** (`UserAuth`) → `widgets`.
* **Многошаговый сценарий, который управляет несколькими фичами?** (`MortgageCalculatorStepCoordinator.vue`) → `widgets` (координатор).
* **Отдельную страницу, доступную по URL?** → `pages`.

# Особенности проекта и стека

### **1. Динамическое управление Layout'ами страниц**

#### **Описание**

Проблема дублирования кода разметки решается через **динамическую привязку layout-компонента к роуту**. Вместо того чтобы каждая страница импортировала и управляла своим layout'ом, нужный layout указывается в `meta`-поле конкретного роута. Корневой компонент `App.vue` считывает это поле и динамически отображает требуемый layout, передавая в него компонент страницы через слот.

#### **Влияние на архитектуру**

* **Разделение ответственности:** Компоненты страниц отвечают только за контент. Логика компоновки и обрамления страницы (хедер, футер, сайдбар) выносится на уровень маршрутизации.
* **Централизованное управление:** Связь "роут-layout" явно объявлена в одном месте — в конфигурации роутера, что делает структуру приложения прозрачной.

#### **Обоснование**

Этот подход делает компоненты страниц чище и более переиспользуемыми. Он упрощает смену layout'ов для целых групп страниц и обеспечивает гибкость и масштабируемость системы в целом, так как добавление новых вариантов разметки не требует правок в компонентах страниц.

-----

### **2. Использование Vue Composables**

#### **Описание**

**Composables** — это функции, использующие Composition API (`ref`, `reactive`, `computed`, `watch`) для инкапсуляции и повторного использования **логики с состоянием (stateful logic)**. Это современный и предпочтительный способ организации переиспользуемой логики во Vue.

#### **Влияние на архитектуру**

* **Логическая декомпозиция:** Сложная логика компонентов разбивается на небольшие, тестируемые и переиспользуемые Сomposable-функции (например, `usePagination`, `useSearch`, `useFormValidation`).
* **Чистая композиция:** В отличие от миксинов, Сomposables не приводят к конфликтам имен и не создают неявных зависимостей. Источник каждого свойства и метода всегда очевиден.

#### **Обоснование**

Composables обеспечивают полную типизацию в TypeScript, легко комбинируются друг с другом и не имеют недостатков предыдущих подходов (миксины, renderless-компоненты). Это делает код более читаемым, поддерживаемым и надежным.

-----

### **3. Координирующий виджет (Dependency Inversion)**

#### **Описание**

Для композиции нескольких независимых виджетов или фичей, которым нужно обмениваться данными, используется паттерн **"Координирующий виджет"**. Это компонент-контейнер, который управляет состоянием и логикой, но не знает о конкретных дочерних компонентах. Он предоставляет **именованные слоты** и передает в них данные и колбэки через `v-slot`.

#### **Влияние на архитектуру**

* **Инверсия зависимостей:** Координатор (высокоуровневый модуль) не зависит от конкретных дочерних виджетов (низкоуровневых модулей). Вместо этого он зависит от **абстракции** — контракта слота. Конкретные реализации подставляются извне (в компоненте-родителе, например, на странице).
* **Контролируемое нарушение FSD:** Композиция виджетов (что запрещено по FSD) происходит осознанно и в одном месте — в шаблоне родителя, который использует координатор. Сами виджеты при этом остаются независимыми.

#### **Обоснование**

Этот подход позволяет создавать сложные, интерактивные блоки из независимых компонентов, сохраняя их слабую связанность и высокую переиспользуемость. Координатор можно повторно использовать с разными наборами дочерних виджетов, что делает архитектуру гибкой.

```vue
<MortgageCalculatorStepCoordinator>
  <template #preliminaryCalculation="{ switchToUserAuth }">
    <PreliminaryCalculation @submit="switchToUserAuth" />
  </template>

  <template #userAuth>
    <UserAuth />
  </template>
</MortgageCalculatorStepCoordinator>
```

-----

### **2. Правило использования `export` и `export default`**

Главное правило: выбор должен отражать **основное назначение файла**.

* **`export default`**: Используется, когда файл предоставляет **одну главную сущность**. Это "главный герой" файла.
    * **Примеры:** Vue-компоненты, Pinia-сторы, классы, конфигурационные объекты.
* **Именованный `export`**: Используется, когда файл предоставляет **набор** вспомогательных, независимых утилит, констант или типов. Это "ящик с инструментами".
    * **Примеры:** `utils.ts`, `constants.ts`, `types.ts`.

> **Примечание:** Смешивать `default` и `named` экспорт в одном файле не рекомендуется для сохранения чистоты и предсказуемости кода.

| Критерий | `export default` | Именованный `export` |
| :--- | :--- | :--- |
| **Основная цель** | Файл для одной главной сущности | Файл как библиотека утилит/констант |
| **Количество на файл**| Только **один** | **Много** |
| **Импорт** | `import AnyName from './file'` | `import { exactName } from './file'` |
| **Аналогия** | "Главный герой" файла | "Набор инструментов" |

### **3. реактивности Vue**

| Метод | Что делает | Когда использовать |
| :--- | :--- | :--- |
| `ref()` | Создает реактивную ссылку на любое значение (примитив или объект). | Для примитивов или когда нужно переприсвоить все значение целиком. |
| `reactive()` | Создает реактивный прокси-объект или массив. | Для структурированных данных (объектов, массивов). Нельзя переприсвоить. |
| `readonly()` | Создает неизменяемый (read-only) прокси для `ref` или `reactive`. | Для безопасной передачи состояния в дочерние компоненты (props). |
| `toRefs()` | Преобразует свойства `reactive` объекта в набор `ref`-ссылок. | Для безопасной деструктуризации `reactive`-объекта без потери реактивности. |

### **4. Промпт для сортировки импортов**

Ты — опытный JavaScript разработчик. Моя задача — привести в порядок импорты в файлах проекта, следуя строгому стилю. Пожалуйста, обрабатывай файлы по одному и сортируй импорты в следующем порядке:

1.  **Внешние зависимости:** Импорты из `vue`, `node_modules` и других внешних библиотек (например, `axios`, `lodash`).
2.  **Внутренние абсолютные импорты из `@/shared`:** Все импорты, начинающиеся с `@/shared/...`.
3.  **Внутренние абсолютные импорты из текущего домена/слоя:** Все импорты, которые относятся к текущему сегменту (например, если файл находится в `@/entities/user`, то сюда относятся импорты `@/entities/user/...`).
4.  **Внутренние импорты из других доменов/слоев:** Все импорты из других абсолютных путей, не попавшие в предыдущие группы (например, `@/features/auth`, `@/widgets/header`).
5.  **Относительные импорты (`./` или `../`):** Все импорты, начинающиеся с точки.

**Правила внутри каждой группы:**

* Сортируй импорты в алфавитном порядке по имени модуля (пути после `from`).

**Важные указания:**

* Не изменяй код файла, кроме секции импортов.
* Не добавляй и не удаляй импорты, только меняй их порядок.
* Всегда оставляй пустую строку между указанными выше группами для визуального разделения.
* Убедись, что после сортировки синтаксис кода остается рабочим.

# Подсказки по TypeScript

`interface` vs `type`

Главное правило: **используйте `interface` для описания формы объектов**, а **`type` — для всего остального** (объединений, псевдонимов для примитивов, сложных комбинаций и т.д.).

-----

### **1. Когда использовать `interface` 🏛️**

Используйте `interface` для описания "контрактов" или "чертежей" объектов и классов. Это их основное предназначение.

**Ключевая особенность:** Интерфейсы можно расширять (`extends`), что идеально подходит для создания иерархии объектов.

**Пример:**
```typescript
// Описываем базовую структуру пользователя
interface User {
  id: number;
  name: string;
}

// Расширяем User, добавляя новые поля для админа
interface Admin extends User {
  permissions: string[];
}

const regularUser: User = { id: 1, name: 'Иван' };
const adminUser: Admin = { id: 2, name: 'Мария', permissions: ['create', 'delete'] };
```

-----

### **2. Когда использовать `type` 🧬**

Используйте `type` (псевдоним типа), когда вам нужно что-то большее, чем просто форма объекта.

**Ключевая особенность:** `type` невероятно гибок. Он может описывать объединения, кортежи, псевдонимы для примитивов и сложные комбинации типов.

**Примеры:**

**1. Для объединений (Union Types) — самый частый случай:**

```typescript
type Status = 'pending' | 'success' | 'error';

let currentStatus: Status = 'pending';
currentStatus = 'success';
// currentStatus = 'loading'; // Ошибка! Такого значения нет в типе Status.
```

**2. Для псевдонимов примитивов (для читаемости):**

```typescript
type UserID = string;
type ProductID = string;

function getUser(id: UserID) { /* ... */ }
function getProduct(id: ProductID) { /* ... */ }
```

**3. Для сложных комбинаций типов:**

```typescript
type ID = number | string;

type ApiResponse = {
  data: any;
} | {
  error: Error;
};
```

------

### **3. Анализ кода: `interface` или `type`**

Давайте проанализируем код, основываясь на этих правилах.

```typescript
import { type ComputedRef } from 'vue';

// 1. GqlCustomerPortalStyle
export interface GqlCustomerPortalStyle {
  email: string;
  __typename: string;
}

// 2. PortalStyle
export interface PortalStyle {
  email: string;
}

// 3. PortalStyleState
export interface PortalStyleState {
  isLoading: ComputedRef<boolean>;
  error: string | null;
  portalStyle: PortalStyle | null;
}
```

#### Вердикт:

Ваш код **полностью соответствует лучшим практикам**\!

1.  **`GqlCustomerPortalStyle` и `PortalStyle`**: Это классические примеры описания формы объекта. Вы описываете, какие поля и какого типа должны быть в объектах, которые приходят от GQL или используются в приложении. Использование `interface` здесь — **идеальный выбор**.

2.  **`PortalStyleState`**: Это также описание формы объекта (состояния). Несмотря на то, что внутри используются объединения (`string | null`, `PortalStyle | null`), сама сущность `PortalStyleState` — это объект. Поэтому использование `interface` здесь тоже **полностью оправдано и является хорошей практикой**.

#### Мог бы здесь использоваться `type`?

Да, для `PortalStyleState` можно было бы использовать и `type`. Это не было бы ошибкой, так как "состояния" редко расширяются через `extends`.

```typescript
// Альтернативный, но тоже корректный вариант для PortalStyleState
export type PortalStyleState = {
  isLoading: ComputedRef<boolean>;
  error: string | null;
  portalStyle: PortalStyle | null;
};
```

**Но ваш текущий вариант с `interface` абсолютно правильный и, возможно, даже более предпочтительный, так как вы последовательно используете один и тот же инструмент для описания всех форм объектов.**

#### Итог и главная рекомендация

* Ваш код уже написан отлично.
* Продолжайте использовать `interface` для описания всех "чертежей" объектов.
* Используйте `type`, когда вам понадобятся объединения (например, `type TButtonType = 'primary' | 'secondary';`).
* **Самое главное — будьте последовательны.** Единообразный стиль в проекте важнее, чем строгие дебаты "interface vs type".
